# Discord UX Spec

Discord上でのbochi出力・インタラクション品質仕様。

## 画面占有計算

### スマホ画面サイズ前提（iPhone 14/15基準）

| パラメータ | 値 |
|-----------|-----|
| 画面幅 | 390px |
| Discordメッセージ領域幅 | ~300px（アバター+パディング除く） |
| フォントサイズ | 16px（Discord iOS default） |
| 1行あたり文字数（日本語） | ~18-20文字 |
| 1行あたり文字数（英数字） | ~38-42文字 |
| 画面高さ（有効領域） | ~600px（ナビ+入力欄除く） |
| 1画面に表示される行数 | ~25行 |
| **1画面に収まる日本語文字数** | **~320-480文字** |

### 占有率の目安

| 文字数 | 画面占有 | 体感 |
|--------|---------|------|
| ~200文字 | 40% | 快適。他のメッセージも見える |
| ~500文字 | 100% | 1スクロールで読める限界 |
| ~1000文字 | 200% | 長い。スクロール必須 |
| ~1900文字 | 400% | 圧迫感。タイムライン埋没 |

## メッセージ長制限

### ルール

<HARD-GATE>
Discord返信は**セクション単位で分割**し、**各メッセージ300文字以内**に収める。

DMにはスレッド機能がないため、文字数で途中カットするのではなく、
出力を意味のあるセクションに分解し、各セクションを個別メッセージとして送信する。

セクション自体が300文字を超える場合は、さらに1情報（1項目）ずつに分割する。
</HARD-GATE>

### 分割フロー

```
[bochi処理完了]
  |
  [出力をセクション単位に分解]
  |
  [各セクションの文字数チェック]
  +-- ≤300文字 → そのまま1メッセージ
  +-- >300文字 → セクション内の項目を1つずつ個別メッセージに分割
  |
  [順番に reply tool で送信]
  → 最初のメッセージのみ通常送信（message_id取得）
  → 2つ目以降は reply_to: 最初のmessage_id で参照チェーン
```

### Mode別のセクション分解

**Mode 1（アイデア膨らまし）:**

| # | セクション | 目安文字数 | 300超時 |
|---|-----------|-----------|---------|
| 1 | 仮説サマリー | ~150 | — |
| 2 | 主要発見（3件） | ~300 | 1件ずつ3メッセージ |
| 3 | ソース（📎リンク2件） | ~100 | — |
| 4 | 次のアクション選択肢 | ~100 | — |

**Mode 2（新聞）:**

| # | セクション | 目安文字数 | 300超時 |
|---|-----------|-----------|---------|
| 1 | 挨拶 | ~30 | — |
| 2 | カテゴリ1: 3記事 | ~200 | 1記事ずつ |
| 3 | カテゴリ2: 3記事 | ~200 | 1記事ずつ |
| ... | （カテゴリごとに1メッセージ） | | |
| N | クロージング | ~50 | — |

**Mode 3（雑談）:** 3-5項目。通常300文字以内に収まる → 1メッセージ。超過時は項目ごと。

**Mode 4（記憶）:** メモ一覧。タイトル+日付のみなので短い。詳細要求時はメモ1件ずつ。

**Mode 5（コンパニオン）:** 1-2件の要約。常に1メッセージで収まる。

## リアクション・ステータスシステム

### 設計原則

1. **ackReaction（access.json）は無効化**（空文字に変更）
2. **bochi自身がreactツールで状況に応じたリアクションを付ける**
3. リアクションは**ステータス表示**として機能する
4. 同じ絵文字を毎回使わない — カテゴリ内からランダム選択

### ステータスフロー

```
メッセージ受信
  |
  [Phase 1: 受信確認] → react: received カテゴリからランダム1つ
  |
  [Phase 2: 処理中]  → react: processing カテゴリからランダム1つ（追加）
  |
  [Phase 3: 完了]    → 処理結果に応じた reply を送信
                       （リアクションはそのまま残る = 処理履歴として機能）
```

### リアクション絵文字カテゴリ

| Phase | カテゴリ | 候補プール | 意味 |
|-------|---------|-----------|------|
| 受信確認 | received | 👀 👁️ 📨 📩 📬 | 「見たゆ」 |
| 調査中 | searching | 🔍 🔎 🧐 📡 🕵️ | 「調べてるゆ」 |
| 発見 | found | 💡 ✨ 🎯 📌 🔖 | 「見つけたゆ」 |
| 処理中 | processing | ⏳ ⚙️ 🔄 💭 🧠 | 「考えてるゆ」 |
| 完了 | done | ✅ 🎉 💫 🌟 ☑️ | 「できたゆ」 |
| エラー | error | ⚠️ ❌ 🔧 | 「うまくいかなかったゆ」 |

### 選択ルール

- 各カテゴリの候補プールから**ランダムに1つ選択**
- 同一メッセージに対して同じカテゴリのリアクションは1つだけ
- 直前のメッセージと同じ絵文字にならないよう1つ前を記憶

### Mode別ステータスフロー

**Mode 1（アイデア膨らまし）— 長い処理:**
```
received(👀) → searching(🔍) → found(💡) → processing(🧠) → reply送信
```

**Mode 2（新聞）— 中程度の処理:**
```
received(📩) → searching(🔎) → reply送信
```

**Mode 3（雑談）— 短い処理:**
```
received(👁️) → reply送信
```

**Mode 4/5（記憶/コンパニオン）— ローカル処理:**
```
received(📬) → reply送信
```

## 長時間処理時のテキストフィードバック

リアクションだけでは伝わりにくい場合、edit_messageで進捗を通知:

```
[リサーチ開始]
  → reply: "調べてるゆ...🔍"（message_id_progress取得）
  |
[3件見つかった]
  → edit_message(message_id_progress): "3件見つけたゆ、もう少し調べるゆ...📡"
  |
[処理完了]
  → reply: 本文（新メッセージ。push通知が届く）
  → delete or そのまま残す（progress message）
```

**重要**: edit_messageはpush通知を発生させない。最終結果は必ず新メッセージで送信する。

## FigJam図の添付

Mode 1 Phase E で OST図やフロー図を生成した場合:

```
[bochi Phase E output]
  |
  [FigJam MCP available?]
  +-- Yes
  |     → generate_diagram で FigJam に図を生成
  |     → get_screenshot でスクリーンショット取得
  |     → reply tool の files パラメータで画像添付
  |     → 本文に「図も作ったゆ 📎」
  |
  +-- No (MCP unavailable)
        → テキストのみで出力（従来通り）
```

### 添付パターン

| 図の種類 | 生成方法 | 添付先 |
|---------|---------|--------|
| OST（機会→解決策→実験） | FigJam generate_diagram | 本文メッセージに添付 |
| リサーチマップ | FigJam generate_diagram | 引用返信に添付 |
| フロー図 | FigJam generate_diagram | 引用返信に添付 |

ファイルパスは `/tmp/bochi-*.png` に一時保存し、送信後に削除。
